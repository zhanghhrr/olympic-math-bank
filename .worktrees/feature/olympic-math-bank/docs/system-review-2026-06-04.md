# 奥林匹克数学题库管理系统 -- 系统性审视报告

> 评估日期：2026-06-04
> 评估视角：开发者 + 教研人员/教师使用者
> 核心问题：系统能否帮助教研人员与老师快速导入题目、选择题目并导出美观的讲义或试卷？

---

## 总评

当前系统在 **OCR 导入流水线** 方面已经建立了强大的技术壁垒，是其核心竞争力。**题目筛选与选择** 功能基本可用但缺乏持久化能力。**导出/组卷** 方面已有工程基础，但在"美观讲义与试卷"的用户体验上距离教研人员的实际需求还有明显差距。

---

## 一、快速导入题目：7.5/10

### 已做好的

| 能力 | 评价 |
|---|---|
| MinerU v4 云端 OCR 识别 | 引擎选型合理，VLM 模型 + 公式/表格识别 |
| SHA256 缓存去重 | 避免重复 API 调用，节省成本 |
| 异步导入 + 前端轮询 | OCR 长任务不阻塞请求，UX 合理 |
| 题目自动拆分（`HybridQuestionIdentifier`） | 按标题 + 序号模式拆分，含章节注入，设计精细 |
| 题型自动检测（填空/选择/解答/计算） | 覆盖主要题型 |
| 自动打标签（关键词 + LLM 兜底） | `knowledge-keywords.ts` 覆盖 100+ 知识点，设计优秀 |
| 公式四层校验（KaTeX 语法 + 启发式 + 交叉 + 双引擎） | 严谨 |
| SimHash 近似去重 | 有效防止重复入库 |
| 导入预览 + 用户纠错 + 确认入库 | 人机协同闭环完整 |
| 自动推断年级/难度 | 有辅助价值 |

### 需要改进的

1. **无 RapidOCR 备用链路**：项目规范中多次提到 `rapidocr-client.ts` 和 Python OCR 脚本作为备用链路，但实际代码库中不存在。一旦 MinerU API 不可用或额度耗尽，系统完全不可导入。建议实现本地 RapidOCR 兜底方案。

2. **公式编辑器在导入预览中的缺失**：预览页可以修改文本，但公式纠错（特别是 OCR 识别错误的 LaTeX）体验未知 -- 是否支持 `VisualLatexEditor` 原地编辑公式？这是 OCR 纠错的核心痛点。

3. **批量导入的"一键通过"**：如果 OCR 质量高（如印刷清晰的标准试卷），用户应该能"全选 → 一键确认导入"，无需逐题检查。当前流程是否支持这个路径？

4. **图片题支持不明确**：`EditableImage` 组件已存在，但 MinerU 返回的图片（几何图形、数阵图等）能否自动关联到题目？导入流水线对此的处理逻辑需确认。

## 二、选择题目：6.5/10

### 已做好的

| 能力 | 评价 |
|---|---|
| 5 维度筛选（搜索/状态/年级/题型/知识标签树） | 覆盖面够用 |
| 知识标签树筛选（含子孙节点） | 实现正确 |
| checkbox 多选 + 全选当前页 | 基本可用 |
| 悬浮购物车工具栏 | 交互流畅，含批量编辑/打标签/导出 |
| 批量编辑（年级/难度） | 有用 |
| 批量打标签（`BatchTagDialog`） | 替换模式，交互清晰 |
| 批量删除 | API 完备 |

### 需要改进的

1. **购物车无法持久化 -- 这是最大的短板**

   当前选中的题目 ID 仅存储在客户端 `Set<string>` 中，刷新页面即丢失。教研人员的真实工作流是：
   - 周三筛选了一批"等差数列"题目放入购物车
   - 周四又找到几道"行程问题"追加进来
   - 周五最终确认后生成试卷

   没有持久化购物车，这个跨会话的工作流完全不可行。**需要在数据库中增加 `QuestionCollection` 模型**（或至少在 `localStorage` 中持久化）。

2. **无"题集/收藏夹/试卷库"概念**

   老师 A 为三年级准备了一套"期中复习卷"，想保存下来下学期复用。当前系统无法做到 -- 没有 `Exam` 或 `Handout` 模型来持久化试卷编排结果。

3. **分页筛选的体验问题**

   当前全选只支持"当前页"。如果筛选结果有 200 题，用户想全选，需要翻 10 页逐页勾选。缺少"全选所有筛选结果（跨页）"的功能。

4. **难度筛选后端已支持但前端未暴露**

   API 支持 `difficulty` 参数，但题目列表页的筛选面板中没有难度下拉框。

5. **无"最近使用"/"高频使用"筛选**

   老师经常需要找"上次用过的那道题"。没有按使用频率或最近查看排序的能力。

## 三、导出美观的讲义/试卷：5.5/10

### 已做好的

| 能力 | 评价 |
|---|---|
| 四种导出格式（JSON/CSV/MD/PDF） | API 完备 |
| React-PDF 服务端 PDF 生成 | 底层链路打通 |
| KaTeX SVG 引擎（自研 641 行） | 技术难度高，公式渲染正确 |
| NotoSansSC 中文字体嵌入 | 中文 PDF 不乱码 |
| A4 打印预览页面 | 拖拽排序、学生/教师双模式 |
| 浏览器打印 + PDF 下载双通道 | 灵活 |
| 主副标题可编辑 | 基本信息定制 |

### 需要改进的（按优先级排列）

1. **讲义模板缺失 -- 最核心的差距**

   当前只有"学生版（仅题目+作答区）"和"教师版（含答案）"两种模式。但**美观的讲义**需要：
   - **知识框架图**：讲义开头展示本讲涉及的知识标签树
   - **例题 + 练习分区**：例题区（带解析）和练习区（纯题目）同页对比
   - **课堂笔记栏**：右侧留出笔记空白区
   - **页眉/页脚**：显示"奥林匹克数学训练讲义 · 第X讲"、页码、日期
   - **封面页**：讲义标题、适用年级、知识点列表、机构 Logo

   这些在当前的 `document.tsx` 中完全没有。

2. **试卷排版的专业性不足**

   - **无双栏排版**：A4 纸单栏排数学题，大量空白浪费。标准试卷通常是双栏。
   - **无密封线/装订线**：正式考试卷需要在左侧留密封线（姓名/班级/学号）
   - **无题型分组标题**：如"一、填空题（每题5分，共20分）"，当前标题块过于简陋
   - **无分数标注**：每道题的分数分值无法设置
   - **无"答案页"分离**：教师版答案直接跟在题目下方，但实际教研需要"题目卷 + 答案卷"分开发放

3. **PDF 下载体验**

   - 当前 `POST /api/export/pdf` 接收的是排版后的 `blocks` 数据（前端组好的），不是题目 ID 列表。这意味着用户必须先进入打印页手动排版，才能下载 PDF。
   - **缺少"一键生成试卷 PDF"**：从题库列表选中 20 题 → 直接下载 PDF（使用默认模板自动排版）。

4. **无试卷边框/水印/Logo**

   项目规范中提到"暖陶土学术风"，但 PDF 输出没有体现品牌设计。正式试卷通常有学校/机构 Logo、密封线边框、页码水印等。

5. **公式在 PDF 中的渲染可能有边缘情况**

   自研的 KaTeX SVG 解析器（`render-katex.ts` 641 行）是风险点 -- KaTeX 版本升级可能导致 HTML 结构变化，SVG 解析失效。需要回归测试覆盖。

6. **xlsx 依赖已安装但无 Excel 导出 API**

   CSV 在 Excel 中打开的中文体验不稳定（虽然加了 BOM），xlsx 库可以直接生成 `.xlsx`，这个功能缺失。

## 四、全局架构问题

### 4.1 缺少题目集合持久化模型

数据库中没有 `QuestionCollection` / `Exam` / `Handout` 模型。建议新增：

```prisma
model QuestionCollection {
  id          String   @id @default(cuid())
  name        String                        // 题集名称
  description String?                       // 描述
  type        String   @default("CART")    // CART | EXAM | HANDOUT
  blocks      String?                      // JSON: 排版块数据（试卷/讲义专用）
  userId      String
  user        User     @relation(fields: [userId], references: [id])
  questions   QuestionCollectionItem[]
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}

model QuestionCollectionItem {
  collectionId String
  collection   QuestionCollection @relation(fields: [collectionId], references: [id], onDelete: Cascade)
  questionId   String
  question     Question           @relation(fields: [questionId], references: [id], onDelete: Cascade)
  sortOrder    Int                // 排序序号
  score        Int?               // 题目分值
  @@id([collectionId, questionId])
}
```

### 4.2 版本快照已有但未被导出利用

`QuestionVersion` 模型已实现版本回滚，但导出/打印时没有"使用历史版本"的选项。如果某题被编辑坏了，用户无法选择导出旧版本。

### 4.3 审核流程与导出未打通

审核通过的题目（`APPROVED`）和草稿（`DRAFT`）在导出时没有区分。教研人员可能不小心把未审核的草稿题导出到正式试卷中。建议导出时默认只包含 `APPROVED` 状态的题目，或给出明确警告。

## 五、优先级建议

| 优先级 | 改进项 | 影响范围 |
|---|---|---|
| **P0** | 购物车持久化（至少 localStorage） | 所有用户的日常工作流 |
| **P0** | 题目集合持久化模型 + "我的题集"页面 | 组卷工作流的闭环 |
| **P1** | 讲义模板（知识框架图 + 例题/练习分区 + 笔记栏 + 封面） | 教研场景的核心价值 |
| **P1** | 双栏排版 + 密封线 + 题型分组标题 + 分值标注 | 试卷的专业性 |
| **P1** | 一键生成 PDF（从题库列表直接导出，默认模板） | 降低使用门槛 |
| **P2** | RapidOCR 本地备用链路 | 系统可用性 |
| **P2** | 导入预览中的公式编辑器集成 | OCR 纠错体验 |
| **P2** | 跨页全选 + 难度筛选前端暴露 | 筛选体验 |
| **P2** | Excel (.xlsx) 导出 | 数据交换 |
| **P3** | 试卷 Logo/水印/边框模板 | 品牌化 |
| **P3** | 答案卷分离 | 考试场景 |

---

## 六、一句话总结

**系统在"OCR 识别 → 自动解析 → 标签匹配 → 入库"这条链路上已经做到了产业级水平，但从"入库的题目"到"老师手中美观可用的讲义/试卷"之间，还缺一个"组卷工作台"产品层。** 当前最紧迫的是题目集合持久化（让老师能保存和复用组卷结果）和讲义/试卷的专业模板（让输出真正"美观可交付"）。
