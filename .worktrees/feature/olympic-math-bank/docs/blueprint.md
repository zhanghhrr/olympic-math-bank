## 1. 系统概述

奥数题库管理系统是深圳桃李未来教育集团的核心数字资产平台，旨在将公司多年积累的数学竞赛教学内容（教材、试卷、练习册）进行**结构化数字化**，实现从纸质资源到智能题库的完整闭环。

### 核心定位

```
纸质教材/试卷  →  OCR自动识别  →  结构化题库  →  智能组卷  →  打印输出/在线发布
   (输入)           (处理)          (存储)        (应用)         (输出)
```

### 解决的业务痛点

| 痛点 | 描述 |
|------|------|
| **内容碎片化** | 教材、试卷分散在各校区教师的个人电脑和纸质文件中，无法统一管理 |
| **出题效率低** | 教师手动从多本教材中翻找题目，拼凑试卷耗时 2+ 小时 |
| **质量不可控** | 各校区独立出题，难度、知识点覆盖全凭个人经验 |
| **知识流失** | 资深教师离职即带走多年积累的教学经验和题目资源 |
| **无数据支撑** | 没有题目使用数据、难度数据，教研决策纯靠拍脑袋 |
| **复用率低** | 大量优质题目沉睡在 PDF 中，从未被有效检索和复用 |

## 4. 核心技术栈

### 4.1 前端

| 技术 | 用途 | 选型理由 |
|------|------|----------|
| **Next.js 14** | 全栈框架 | SSR/SSG 混合渲染，API Routes，Turbopack 构建 |
| **React 18** | UI 框架 | 组件化开发，丰富的生态 |
| **Tailwind CSS** | 样式方案 | 原子化 CSS，快速开发，设计师友好 |
| **TypeScript** | 类型系统 | 大型项目必需的代码可维护性保障 |

### 4.2 后端

| 技术 | 用途 | 选型理由 |
|------|------|----------|
| **Next.js API Routes** | HTTP 接口 | 与前端同构，减少上下文切换 |
| **Prisma** | ORM | 类型安全的数据库操作，自动生成 TypeScript 类型 |
| **NextAuth.js** | 认证 | 开箱即用的 OAuth/邮箱登录方案 |
| **SQLite** | 数据库 | 零配置，单文件部署，适合中小规模（<100万条） |

### 4.3 OCR 引擎

| 技术 | 用途 | 选型理由 |
|------|------|----------|
| **RapidOCR** | 中文 OCR 识别 | 基于 PaddleOCR PP-OCRv4，轻量、离线、中文精度高 |
| **MinerU** | 复杂排版 OCR | 版面分析能力强，适合公式+图文混排 |
| **Docling** | 版面解析 | IBM 开源的文档理解框架，结构化输出 |
| **PyMuPDF** | PDF 渲染 | PDF → 图片的高质量渲染引擎 |
| **Python 3** | OCR 脚本语言 | AI/OCR 生态最完善的语言 |

### 4.4 为什么选择 SQLite 而非 PostgreSQL？

| 因素 | SQLite | PostgreSQL |
|------|--------|-----------|
| 部署复杂度 | 零配置，单文件 | 需要独立服务进程 |
| 并发能力 | 适合 <100 并发用户 | 适合高并发 |
| 数据量 | 适合 <100 万条 | 适合亿级 |
| 备份 | 复制文件即可 | 需要 pg_dump |
| **当前需求匹配** | ✅ 完全满足 | 过度设计 |

> 📌 **升级路径：** 当题库 >50 万题或并发用户 >100 时，可通过 Prisma 无缝迁移至 PostgreSQL。

## 5. 模块设计

### 5.1 OCR 导入链路

```
  PDF 上传
     │
     ▼
  ┌──────────────────────────────────────────────┐
  │  PyMuPDF 渲染                                 │
  │  PDF → 多页 PNG 图片 (400 DPI)                │
  └──────────────────┬───────────────────────────┘
                     │
                     ▼
  ┌──────────────────────────────────────────────┐
  │  水平线增强预处理                              │
  │  enhance_underlines()                         │
  │  检测细水平线 → 加粗 → 提升 OCR 下划线识别率    │
  └──────────────────┬───────────────────────────┘
                     │
                     ▼
  ┌──────────────────────────────────────────────┐
  │  Docling 版面分析 + RapidOCR 文字识别          │
  │  输出：Markdown（含 LaTeX 公式）               │
  └──────────────────┬───────────────────────────┘
                     │
                     ▼
  ┌──────────────────────────────────────────────┐
  │  TypeScript 后处理                             │
  │  · cleanQuestionContent()  9 步清洗流水线     │
  │  · inferBlankPlaceholders()  下划线恢复       │
  │  · detectQuestionType()     题型分类          │
  │  · splitQAInBlock()         答案分离          │
  │  · autoMatchKnowledgeTags() 自动打标          │
  └──────────────────┬───────────────────────────┘
                     │
                     ▼
  ┌──────────────────────────────────────────────┐
  │  Prisma 数据入库                               │
  │  Question + KnowledgeTag + ImportJob 联动      │
  └──────────────────────────────────────────────┘
```

### 5.2 题目管理模块

- **CRUD 操作：** 完整的创建、编辑、删除、列表分页
- **搜索过滤：** 按年级、题型、难度、知识点、关键词多维检索
- **版本控制：** QuestionVersion 表记录每次修改，支持历史查看与回滚
- **审核工作流：** Draft → Pending → Approved / Rejected，含审核意见
- **图片管理：** 题目配图独立存储，与 OCR 结果关联

### 5.3 知识标签体系

```
Level 1: 模块 (Module)        计算模块        几何模块        应用模块
                                  │               │               │
Level 2: 专题 (Topic)     ┌──────┼──────┐        ...             ...
                       整数    小数    分数
                        │
Level 3: 子专题         整数加减    整数乘除
(Subtopic)              │
                        │
Level 4: 知识点         整数加法运算
(Knowledge)            │
                       │
Level 5: 技能          加法横式    加法竖式
(Skill)
```

每个标签有唯一的 `code` 编码（如 `计算模块-整数-整数加减-整数加法运算-加法横式`），支持精确匹配和模糊搜索。

### 5.4 组卷打印模块

- **DocumentBlock 块状结构**：类似 Notion 的拖拽式排版
- **块类型：** 大标题、小标题、题目内容、手动分页符
- **打印控制：** CSS `@media print` + `page-break-after/inside`
- **双模式：** 学生版（预留作答空白）+ 教师版（含答案和解析）
- **排版总览：** 左侧可交互大纲，点击跳转，打印时自动隐藏

## 7. 关键数据流

### 7.1 题目导入全流程

```
教师上传 PDF
     │
     ▼
  [前端] POST /api/import/ocr
     │  → 创建 ImportJob (status: PENDING)
     ├─→ 存储原始 PDF 至 uploads/
     │
     ▼
  [后端] rapidocr-client.ts
     ├─→ 调用 Python 脚本 pdf2md_rapidocr.py
     ├─→ PDF → 图片 → 预处理 → OCR → Markdown
     │
     ▼
  [后端] question-identifier.ts
     ├─→ splitContent()      题目分块
     ├─→ cleanQuestionContent()  9步清洗
     ├─→ detectQuestionType()    题型分类
     ├─→ convertToQuestions()    格式标准化
     │
     ▼
  [后端] import-to-db.ts
     ├─→ 创建 Question 记录 (status: DRAFT)
     ├─→ autoMatchKnowledgeTags()  自动打标
     ├─→ 更新 ImportJob (status: COMPLETED)
     │
     ▼
  [前端] 展示导入结果列表
     → 教师审核 → APPROVED / REJECTED
```

### 7.2 组卷打印流程

```
教师选择题目 → 加入组卷车
     │
     ▼
  /dashboard/print
     ├─→ 加载选中的题目列表
     ├─→ 构建 DocumentBlock[] 块状结构
     ├─→ 教师拖拽调整顺序
     ├─→ 插入标题/分页符
     │
     ▼
  预览与打印
     ├─→ 学生版：渲染题目内容 + 作答空白区域
     ├─→ 教师版：渲染题目内容 + 答案 + 解析（红蓝标注）
     └─→ window.print()  →  CSS @media print 控制分页
```

## 8. 安全与权限

### 8.1 三级角色体系

| 角色 | 权限范围 |
|------|----------|
| **Admin** | 全部权限：用户管理、系统配置、题目删除、批量操作 |
| **Reviewer** | 审核权限：审核题目（Approve/Reject）、查看所有题目 |
| **Editor** | 编辑权限：创建/编辑题目、导入 PDF、查看自己的题目 |

### 8.2 安全措施

- **认证：** NextAuth.js，支持邮箱密码登录 + OAuth
- **授权：** API Route 层面基于 UserRole 的中间件校验
- **数据隔离：** Editor 只能操作自己创建的题目，Reviewer/Admin 可查看全部
- **输入校验：** API 层对所有用户输入进行类型与长度校验
- **环境变量：** 敏感配置（DATABASE_URL、AUTH_SECRET）通过 .env.local 注入，不提交仓库
